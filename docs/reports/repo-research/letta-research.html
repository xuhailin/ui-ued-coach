Letta / MemGPT

Letta（原 MemGPT）是一个开源 Python 服务端与 API 平台，用来构建带长期记忆、工具执行、模型/供应商路由和生产级可观测性的有状态 LLM Agent。

核心概览

它是什么

它不是一个单纯的“记忆库”，而是一整套 Agent 运行时和服务层。仓库里包含 FastAPI 服务、CLI 入口、Agent 循环、记忆模型、工具执行、MCP 集成、模型供应商适配、持久化模型、数据库迁移、测试、Docker 资产、OpenAPI/Fern API 产物和可观测性接入。

适合谁

适合正在做有状态 AI Agent、带记忆的助手、会调用工具的 Agent、多 Agent 工作流，或需要持久化、流式输出、SDK 和多模型支持的内部 Agent API 的团队。

为什么值得看

它把长期运行 Agent 的难点打包在一起：记忆块、归档/召回状态、工具、沙箱、供应商抽象、API 面、迁移和调用链追踪。即使不整套采用，也很适合作为参考架构。

什么时候不适合

如果只是想做一个很小的嵌入式 helper library 或低复杂度聊天机器人脚手架，它会显得太重。它的服务化架构、数据库/迁移面和工具执行模型，超过了很多原型项目的需要。

事实记忆

仓库身份

README.md 将项目命名为 'Letta (formerly MemGPT)'，并描述为一种构建具备高级记忆、可随时间学习和自我改进的 AI 的方式。

主要语言

Python 3.11+ 项目，包含 pyproject.toml、uv.lock；letta/ 下有 536 个 Python 源文件，tests/ 下有 166 个 Python 测试文件。

运行形态

Typer CLI 入口默认通过 letta.main:app 启动服务；REST API 使用 FastAPI；Docker 与 compose 文件支持以 Postgres/pgvector 部署服务。

API 范围

fern/openapi.json 标识 'Letta API' 版本 1.0.0，共 239 个路径。主要标签包括 agents、tools、sources/folders、conversations、runs、blocks、MCP servers、providers、jobs、messages 和 archives。

持久化

SQLAlchemy/SQLModel 模型位于 letta/orm；alembic/versions 包含 167 个迁移；compose.yaml 使用 ankane/pgvector 作为 PostgreSQL 向量存储。

记忆模型

letta/schemas/memory.py 定义核心记忆块、文件块、上下文窗口概览，以及支持 git 的记忆渲染路径。

工具系统

工具执行通过工厂模式覆盖 core、memory、file、builtin、MCP、多 Agent 和沙箱执行器；services/tool_sandbox 中出现 local、E2B 和 Modal 沙箱路径。

供应商支持

pyproject.toml 和 letta/llm_api 中包含 OpenAI、Anthropic、Azure、Bedrock、Google Gemini/Vertex、Groq、Mistral、DeepSeek、Ollama、OpenRouter、Together、vLLM、xAI、Z.ai 等适配器。

质量信号

仓库包含 unit/integration/docker/migration 工作流、ruff/ty 配置、pytest 配置、安全策略、AI 贡献策略、Apache 许可证和较大的测试目录。

架构

flowchart TD
  User[CLI / Python SDK / TS SDK / REST clients] --> API[FastAPI REST API\nletta/server/rest_api]
  User --> CLI[Typer CLI\nletta/main.py]
  CLI --> API
  API --> Server[SyncServer facade\nletta/server/server.py]
  Server --> Managers[Domain managers\nAgent / Message / Block / Tool / Source / Run]
  Managers --> ORM[SQLAlchemy ORM + Alembic\nletta/orm + alembic]
  ORM --> DB[(Postgres / pgvector\nSQLite in tests/dev)]
  Managers --> Memory[Memory system\nblocks / archival / recall / git memory repo]
  Memory --> MemRepo[Object-backed Git memory\nservices/memory_repo]
  Server --> AgentLoop[AgentLoop factory\nV2 / V3 / sleeptime / voice]
  AgentLoop --> Prompt[Prompt + context assembly\nPromptGenerator / Memory.compile]
  AgentLoop --> LLM[LLMClient + provider adapters\nOpenAI / Anthropic / Gemini / Ollama / etc.]
  AgentLoop --> Tools[ToolExecutionManager]
  Tools --> Builtin[Core / memory / files tools]
  Tools --> MCP[MCP tools + servers]
  Tools --> Sandbox[Sandbox execution\nLocal / E2B / Modal]
  AgentLoop --> Streaming[Streaming responses + runs + steps]
  Server --> Observability[OTel / Datadog / Sentry / provider traces]
  API --> OpenAPI[Fern OpenAPI + SDK generation]

Letta 更适合被理解为一个分层 Agent 平台：API/SDK 客户端调用 FastAPI routers，routers 委托给 SyncServer 门面，门面协调领域 managers，managers 通过 ORM/storage 持久化，而 Agent loops 负责编排 prompt、记忆、模型调用、工具、流式输出、runs 和 traces。

采用建议

搭建复杂度

中到高。quickstart 可以走 hosted API/SDK，但自托管会涉及 Docker 或 Python 环境、Postgres/pgvector、迁移、供应商凭证和沙箱设置。

集成路径

产品团队可以从 README.md 的 Letta API 与 SDK 示例开始。平台团队可以先本地跑 Docker compose，查看 fern/openapi.json 的 API resources，再用 memory blocks 和一组受控工具试点一个 Agent。

扩展点

自定义工具、MCP servers、provider configuration、sandbox config、memory blocks、sources/folders、archives、prompts/templates 和 provider trace backends。

学习曲线

如果要改内部实现，学习曲线高；如果只用 API，中等。manager/schema/ORM 分层比较一致，但领域本身很宽。

建议的 PoC

创建一个带 persona/human memory blocks 的 Agent，发送消息，修改记忆，挂载一个 source/folder，运行一个安全工具，检查 run/step traces，并测试服务重启后的持久化。
