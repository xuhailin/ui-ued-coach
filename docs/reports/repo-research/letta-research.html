Executive Snapshot

What it is

A full agent runtime and service layer, not just a memory library. The repo contains a FastAPI server, CLI entrypoint, agent loops, memory models, tool execution, MCP integration, provider adapters, persistence models, migrations, tests, Docker assets, OpenAPI/Fern API artifacts, and observability wiring.

Who it is for

Teams building stateful AI agents, memory-backed assistants, tool-using agents, multi-agent workflows, or an internal agent API that needs persistence, streaming, SDKs, and provider flexibility.

Why it matters

It packages the hard parts around long-lived agents: memory blocks, archival/recall state, tools, sandboxing, provider abstraction, API surface, migrations, and traces. That makes it a useful reference architecture even when not adopted wholesale.

When not to use it

Avoid using it as a tiny embedded helper library or a low-complexity chatbot scaffold. The service-oriented architecture, database/migration surface, and tool execution model are more than many prototypes need.

Fact Memory

Repository identity

README.md names the project 'Letta (formerly MemGPT)' and describes it as a way to build AI with advanced memory that can learn and self-improve over time.

Primary language

Python 3.11+ package with pyproject.toml, uv.lock, 536 Python source files under letta/, and 166 Python test files under tests/.

Runtime shape

Typer CLI entrypoint launches a server by default via letta.main:app; FastAPI is used for the REST API; Docker and compose files support service deployment with Postgres/pgvector.

API surface

fern/openapi.json identifies 'Letta API' version 1.0.0 with 239 paths. Major tags include agents, tools, sources/folders, conversations, runs, blocks, MCP servers, providers, jobs, messages, and archives.

Persistence

SQLAlchemy/SQLModel models live in letta/orm; alembic/versions contains 167 migrations; compose.yaml uses ankane/pgvector for PostgreSQL vector storage.

Memory model

letta/schemas/memory.py defines core memory blocks, file blocks, context window overview, and git-enabled memory rendering paths.

Tooling

Tool execution uses a factory over core, memory, file, builtin, MCP, multi-agent, and sandboxed executors; local, E2B, and Modal sandbox paths appear in services/tool_sandbox.

Provider support

pyproject.toml and letta/llm_api include adapters for OpenAI, Anthropic, Azure, Bedrock, Google Gemini/Vertex, Groq, Mistral, DeepSeek, Ollama, OpenRouter, Together, vLLM, xAI, Z.ai, and others.

Quality signals

The repository includes unit/integration/docker/migration workflows, ruff/ty config, pytest config, a security policy, AI contribution policy, Apache license, and a large tests directory.

Core Design Philosophy

  • Stateful agents are modeled as service-owned entities rather than transient prompt calls.

    Agents, messages, blocks, tools, sources, archives, runs, steps, and provider traces are all first-class resources with managers, ORM models, schemas, and API routes.

Architecture

flowchart TD
  User[CLI / Python SDK / TS SDK / REST clients] --> API[FastAPI REST API\nletta/server/rest_api]
  User --> CLI[Typer CLI\nletta/main.py]
  CLI --> API
  API --> Server[SyncServer facade\nletta/server/server.py]
  Server --> Managers[Domain managers\nAgent / Message / Block / Tool / Source / Run]
  Managers --> ORM[SQLAlchemy ORM + Alembic\nletta/orm + alembic]
  ORM --> DB[(Postgres / pgvector\nSQLite in tests/dev)]
  Managers --> Memory[Memory system\nblocks / archival / recall / git memory repo]
  Memory --> MemRepo[Object-backed Git memory\nservices/memory_repo]
  Server --> AgentLoop[AgentLoop factory\nV2 / V3 / sleeptime / voice]
  AgentLoop --> Prompt[Prompt + context assembly\nPromptGenerator / Memory.compile]
  AgentLoop --> LLM[LLMClient + provider adapters\nOpenAI / Anthropic / Gemini / Ollama / etc.]
  AgentLoop --> Tools[ToolExecutionManager]
  Tools --> Builtin[Core / memory / files tools]
  Tools --> MCP[MCP tools + servers]
  Tools --> Sandbox[Sandbox execution\nLocal / E2B / Modal]
  AgentLoop --> Streaming[Streaming responses + runs + steps]
  Server --> Observability[OTel / Datadog / Sentry / provider traces]
  API --> OpenAPI[Fern OpenAPI + SDK generation]

Letta is best read as a layered agent platform: API/SDK clients call FastAPI routers, routers delegate to a SyncServer facade, the facade coordinates domain managers, managers persist through ORM/storage, and agent loops orchestrate prompts, memory, model calls, tools, streaming, runs, and traces.

Weaknesses / Risks

  • High adoption complexity

    Letta is a service platform with many moving pieces: Postgres/pgvector, migrations, provider credentials, tool sandboxing, background scheduling, API auth/middleware, OpenAPI/SDK artifacts, and observability. This is powerful but not lightweight.

    High
    compose.yamlDockerfilepyproject.tomlletta/server/rest_api/app.pyalembic/versions/
  • Security surface is broad because agents can execute tools

    Tool execution supports local subprocesses, virtualenv setup, fetched credentials, environment variable injection, MCP, E2B, and Modal. Teams adopting it need strict sandbox and credential policies.

    High

Adoption Guide

Setup complexity

Medium to high. The quickstart can use hosted API/SDK paths, but self-hosting involves Docker or Python environment setup, Postgres/pgvector, migrations, provider credentials, and sandbox settings.

Integration path

For a product team, start from the Letta API and SDK examples in README.md. For a platform team, run Docker compose locally, inspect API resources with fern/openapi.json, then pilot one agent with memory blocks and a small controlled tool set.

Extension points

Custom tools, MCP servers, provider configuration, sandbox config, memory blocks, sources/folders, archives, prompts/templates, and provider trace backends.

Learning curve

High if modifying internals; medium if using the API. The manager/schema/ORM layering is consistent, but the domain is broad.

Suggested proof of concept

Create one agent with persona/human memory blocks, send messages, mutate memory, attach a source/folder, run one safe tool, inspect run/step traces, and test persistence across server restart.
